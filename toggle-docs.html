Toggle

A switch control for binary on/off settings that take effect immediately — no save step required. Three variants support different levels of labelling: bare, labeled, and with description.

Anatomy

The toggle is built from a bordered pill track and a sliding thumb. When on, a check icon appears inside the thumb to reinforce the active state beyond colour alone.

Usage

Toggles are for settings that turn on or off instantly — no confirmation or save button needed. Avoid using them when the change requires a form submit.

Use when
  • The setting takes effect immediately on flip (e.g. "Dark mode")
  • There are exactly two states: on and off
  • The toggle lives in a settings panel, a preferences page, or a table action row
  • Screen space is tight and a full checkbox row would feel heavy
Avoid when
  • The change needs a "Save" button — use a Checkbox instead
  • Multiple independent items can be selected — use Checkboxes
  • There are more than two states — use Radio Group or Select
  • You need to confirm a destructive action — use a Dialog

Best practices

Do
Give every toggle a clear, descriptive label so users know exactly what they're controlling.
Don't
Don't render toggles without labels — users have no way to know what each switch controls.

Setting saved automatically

Do
Use a toggle when the setting auto-saves. Add a caption like "saved automatically" when that behaviour isn't obvious.
Don't
Don't pair a toggle with a save button. If the change isn't immediate, use a checkbox instead.

Variants

Three variants cover different labelling needs. The label can appear to the right (default) or left of the track.

Labeled — the standard toggle. A text label sits to the right and the entire row is a click target. Use for settings lists and preference panels where a single line of context is sufficient.

With description — adds a secondary description line below the label. Use when the label alone is not self-explanatory — especially for security settings, billing options, or features that have consequential side effects.
Bare — just the track and thumb, no label. Use only where the surrounding UI makes the purpose unambiguous — for example, an "Enabled" column in a dense settings table where each row already has a row label.

Content guidelines

Toggle labels describe the feature being controlled — not the action of toggling.

Label writing

  • Describe the feature, not the action. Write "Email notifications" not "Enable email notifications" — the toggle's position communicates on/off already.
  • Keep labels short — 2–5 words is ideal. If you need more context, use the with-description variant.
  • Use sentence case. "Dark mode", not "Dark Mode" or "DARK MODE".
  • Avoid negatives. "Show tips" is clearer than "Hide tips". Negative labels are confusing when the toggle is off: does off mean tips are shown or hidden?

Description writing

  • Keep descriptions to 1–2 sentences. Explain the effect or consequence — don't repeat the label.
  • Use plain language. Assume the reader is not technical.
  • Don't include "click to" or "toggle to" — the control communicates the interaction model itself.

Accessibility

Toggles must be keyboard-operable and announce their state to screen readers.

  • Use a real <input type="checkbox"> wired to a <label>. This gives keyboard focus and space-bar toggling for free, and screen readers announce "checked / unchecked".
  • Or use role="switch" with aria-checked="true|false" on a custom element. The switch role reads "on / off" in modern screen readers — a clearer description of a toggle than "checked".
  • Always provide a visible label. Don't rely on tooltips or title attributes alone — they are not accessible to keyboard users.
  • The check icon and thumb position communicate state independent of colour — satisfying WCAG 1.4.1 (Use of Colour).
  • Disabled toggles should use aria-disabled="true" rather than the native disabled attribute when you still need the element to receive focus, so screen reader users can discover why it is unavailable.
  • Focus indicator — the native input's focus ring is preserved. Never suppress it with outline: none without providing an equivalent custom focus style.

Accessibility

Requirements for building an accessible toggle.

  • Use <input type="checkbox"> paired with a <label>, or a custom element with role="switch" and aria-checked.
  • The space bar must toggle the switch. Native checkboxes handle this automatically.
  • The check icon and thumb position communicate state independently of colour — satisfying WCAG 1.4.1.
  • Provide an accessible name via a visible label, aria-label, or aria-labelledby.
  • When using the with-description variant, connect the description with aria-describedby.
  • Use aria-disabled="true" instead of disabled if the control needs to remain focusable.
