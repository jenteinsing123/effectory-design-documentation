What is the Engage Design System?

A single source of truth that connects design decisions to production components.

The Engage Design System is a shared library of components, patterns, design tokens, and usage guidelines built specifically for Effectory's Engage product — the platform that powers employee feedback and engagement surveys across organisations.

It bridges two worlds: the Figma component library that designers work in and the Angular component library that developers ship from. Both sides share the same vocabulary — the same token names, the same component names, the same interaction patterns — so decisions made in design translate directly into code without re-interpretation at every handoff.

The practical benefit is speed and consistency. Teams don't rebuild common UI primitives for each feature. They reach for a tested, accessible component, configure it with props, and ship. When a token changes — say, the brand colour shifts — it propagates everywhere at once, in both Figma and the browser.

Who it's for

The system serves two primary audiences, with a third benefiting alongside them.

Designers

The Figma component library contains every component with anatomy diagrams, usage guidelines, and variant breakdowns. All colour, spacing, and typography values are bound to Figma variables that map 1:1 to CSS custom properties.

Developers

The Angular component library exposes each component via an eff-* selector. Each component page includes a Developer tab with a live example, the full Angular input/output API table, and copy-ready code snippets.

Product Managers

PMs reviewing interaction patterns will find the component pages useful. The usage guidelines describe intended behaviour, edge cases, and when not to use a component — context that saves back-and-forth during review.

How it's organised

Two layers: Foundations first, then Components built on top of them.

Foundations

Foundations define the raw material: colour tokens, typography scale, spacing scale, motion (duration and easing), iconography, shadow levels, border radii, and the grid system. Every component is built exclusively from foundation tokens — no component introduces a raw value that isn't already in the token set.

Components

Components are grouped into functional categories:

Each component page

Every component page is split into two tabs. The Design tab covers anatomy, usage guidelines, variants, interactive states, and the component-level token list. The Developer tab provides a live rendered example, the full Angular @Input / @Output API table, and code snippets ready to paste.

Tech stack

Angular components, CSS custom properties, and Figma as the design source of truth.

Getting help

The system is a living product — questions, gaps, and contributions are all welcome.

If you're unsure how a component should behave or whether one exists for your use case, the Contributing page explains how to raise a question, file a bug, or propose something new. The DS team reviews proposals in a bi-weekly sync open to all product team members.

For a full record of what has changed, see the Changelog. It covers every release with new additions, breaking changes, and fixes.

Where a gap exists between what is designed in Figma and what is shipped in code — for example, a variant that is specced but not yet built — the relevant component page will include a ⚠️ callout explaining the discrepancy and the expected resolution timeline. Nothing is silently missing.
