Upgrade Button

A high-emphasis call-to-action that points users at a paid plan, premium add-on, or upsell flow. Its yellow badge and amber border make it stand out from regular buttons so an upgrade prompt never reads as a generic action.

Anatomy

Three layers — a bordered container, a yellow icon badge that anchors the upgrade meaning, and the label. The arrow-up icon and badge are baked into the component; you don't choose an icon.

1Container
2Badge + arrow
3Label
1
Container — white background with an amber border (border/highlight/base) and a soft action shadow. The amber border is what differentiates an upgrade button from any other secondary button at a glance.
2
Badge + arrow — a 16×16 yellow circle (bg/highlight/base) wrapping a 12×12 arrow-up icon. Always present; never replaced with another icon. The badge is what makes the button feel like an upgrade rather than a navigation action.
3
Label — Poppins Medium 14, content/secondary. Keep the label short and outcome-focused: "Upgrade", "Upgrade plan", "Go Pro" — not "Click to upgrade your account now".

Usage

Use the Upgrade Button only for actions that take the user toward a paid plan or premium feature. For every other call-to-action, reach for a regular Button.

Use when
  • Linking to a pricing page, plan picker, or upgrade modal
  • Unlocking a premium feature inline ("Upgrade to enable benchmarks")
  • Promoting an add-on from inside a free or trial experience
Avoid when
  • The action isn't tied to billing — use a normal Button
  • You need more than one CTA in a group — pair an upgrade button with secondary or tertiary buttons, not with another upgrade button
  • The action could be destructive or irreversible

Best practices

Do
Use a short, outcome-focused label that names what the user gets — "Upgrade", "Upgrade plan", "Go Pro".
Don't
Don't pad the label with filler. The button is meant to be glanceable; long labels break the row and dilute the CTA.
Do
Pair the upgrade button with a low-emphasis dismiss action so the upsell is easy to ignore but easy to accept.
Don't
Never put two upgrade buttons next to each other. The treatment is loud on purpose; doubling it dilutes its meaning.
Do
Always ship the yellow badge with the arrow-up icon. The badge is the recognisable signature of the upgrade pattern.
Don't
Don't drop the badge or swap the arrow for another icon. Without it, the button reads as a regular secondary button with an unusual border colour.

Variants

There is a single visual variant. The upgrade pattern is intentionally restricted — one shape, one badge, one label — so it stays unmistakable wherever it appears.

Default — a single 36-px tall button with white background, amber border, yellow badge, and short label. No size, tone, or icon variants.

States

Three states. The border darkens on hover and pressed; pressed also fills the background with a soft amber tint to confirm the click.

Accessibility

→
Use a real <button> or <a>. If the upgrade button triggers a modal or in-app action, use <button>. If it navigates to a pricing page, use an <a href>.
→
The badge is decorative. Don't add an aria-label or alt text to the icon — the label tells the screen-reader user where the button leads. Adding "arrow up" announces noise.
→
Don't rely on yellow alone. The button's meaning is carried by the label; users with low colour vision should still understand the action without seeing the badge.
→
Keep focus visible. A 2-px brand outline appears on :focus-visible — don't override it with outline:none.

Usage

Default

<button type="button" class="btn-upgrade">
  <span class="btn-upgrade-badge"><i data-icon="arrow-up"></i></span>
  Upgrade
</button>

As a link

If clicking the button navigates to a pricing or plans page, use an anchor instead of a button. Same classes, same markup — just <a href> on the outside.

See plans 
<a href="/billing/plans" class="btn-upgrade">
  <span class="btn-upgrade-badge"><i data-icon="arrow-up"></i></span>
  See plans
</a>

CSS classes

CLASS DESCRIPTION
btn-upgrade The base upgrade button. Apply to a <button> or <a>. Sets the layout, the white container, the amber border, and the action shadow. Always wrap the label and badge inside.
btn-upgrade-badge The yellow circle that wraps the arrow icon. Always include it as the first child of .btn-upgrade. Don't restyle it on a per-instance basis.
<i data-icon="arrow-up"> The arrow-up glyph picked up by icons.js. Sized to fill the badge — don't swap it for another icon.
is-hover Forces hover styling. Use only in static documentation/state previews — never in production.
is-pressed Forces pressed styling. Use only in static documentation/state previews.

Accessibility

→
Pick the right element. <button type="button"> for in-app triggers (open a modal, fire a request); <a href> when the click navigates to another page. Don't fake one with the other.
→
The icon is decorative — don't label it. The badge isn't pronounced; the label carries the meaning. Adding aria-label="arrow up" creates noise on every announcement.
→
Don't rely on yellow alone. Make sure the label text already communicates the action ("Upgrade", "Go Pro") so users who can't perceive the badge still understand it.
→
Keep the focus ring. The component's :focus-visible outline is the only feedback keyboard users get when they tab onto the button.
