Button Big

A larger, high-emphasis button designed for prominent calls-to-action. Use it at key decision points in sequence maps, onboarding flows, and page footers where the action deserves more visual weight than a standard button provides.

Anatomy

The button has three parts. The label is always required; icons are optional and can appear on either side.

1Container
2Icon (optional)
3Label
1
Container — the outer shape carrying the fill, border, corner radius (12px), and shadow. Taller and rounder than a standard button to signal a primary page-level action.
2
Icon (optional) — a 16×16px icon placed left or right of the label. A right-facing arrow or chevron is most common to convey forward momentum.
3
Label — always required. Use action-oriented language: "Get started", "Continue", "Go to overview". Keep it to 1–4 words.

Usage

Button Big is for high-emphasis, page-level calls-to-action. It is not a drop-in replacement for the standard button — reserve it for moments where the action is the main point of a section.

Use when
  • ✓ The button is the primary action at the end of a sequence map step
  • ✓ Placed in a page footer or section CTA where it should draw the eye
  • ✓ There is only one main action on the screen at a time
  • ✓ The context calls for more visual presence than a standard button provides
Don't use when
  • ✗ Inside forms, toolbars, or dialogs — use a standard Button instead
  • ✗ Multiple Button Bigs appear on the same screen — they lose their emphasis
  • ✗ The action is destructive — use a standard Danger button with a confirmation dialog
  • ✗ Space is limited — it needs enough room to breathe

Variants

Two variants cover the full range of emphasis needed for large call-to-action placements.

The strongest call-to-action on a screen. Use for the main forward action in a sequence map step or the leading action in a footer. Only one primary Button Big per view.
A supporting action alongside a primary Button Big — for example "Skip" or "Learn more" next to "Get started". Never use a secondary Button Big without a primary one in the same area.

Content guidelines

Because Button Big carries more visual weight, every word matters more.

Label writing

Do
Use an action verb and be specific about what happens next.
Don't
Don't use vague language or wrap the label across multiple lines.

Icon use

An icon is most useful when it reinforces direction or action type. A right-pointing arrow (arrow-right or chevron-right) placed after the label signals progression. Avoid icons that repeat the meaning of the label — the icon should add clarity, not noise.

Accessibility

→
Use a native <button> element. It gets keyboard focus, Enter/Space activation, and correct role for free. Never use a <div> or <span>.
→
Label must be self-explanatory out of context. Screen reader users navigate buttons by list. "Get started" is fine; "Click here" is not.
→
Disabled buttons need a reason. Add a tooltip or visible hint explaining why the button is disabled, and consider aria-disabled="true" over the native disabled attribute so keyboard users can still reach the button and receive the explanation.
→
Decorative icons need aria-hidden. If an icon sits next to a visible label, add aria-hidden="true" to the SVG so screen readers don't read it twice.

Accessibility

→
Always use a native <button>. Do not use <div> or <a> unless the element actually navigates to a URL. Native buttons receive focus and keyboard activation for free.
→
Prefer aria-disabled over disabled. The native disabled attribute removes the button from the tab order. Use aria-disabled="true" so keyboard users can still focus the button and receive an explanation via tooltip or helper text.
→
Add aria-hidden to decorative icons. Icons placed alongside a visible label should have aria-hidden="true" so screen readers don't announce them twice.
