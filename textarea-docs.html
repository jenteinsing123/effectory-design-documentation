Textarea

A multi-line text input for longer free-form content — descriptions, comments, notes, and any value that doesn't fit on a single line.

Anatomy

The textarea is built from a label, a resizable container, and an optional error message.

1Label
4Resize handle
2Container
3Placeholder
1
Label — optional text above the field that names the expected input. Required for accessibility unless a visible heading or surrounding context provides a programmatic label.
2
Container — the scrollable text area. Its height is set by the author and can be increased by the user via the resize handle. Border color changes to communicate interaction state.
3
Placeholder — hint text shown when the field is empty. Keep it short and format-oriented (e.g. "Add a description…"). Placeholder text disappears the moment the user starts typing.
4
Resize handle — allows users to resize the field vertically to see more of their content. Never disable it unless the height is determined entirely by the content (auto-grow).

Usage

Use a Textarea when the expected input spans multiple lines or needs space to breathe.

Use when
  • The value is multi-line — descriptions, comments, notes, feedback
  • You can't predict how long the input will be
  • The user benefits from seeing their full input while typing
Avoid when
  • The expected value fits on one line — use Text Field instead
  • You need structured multi-value input — use a list component or repeater
  • The field is inside a compact table cell — consider an inline text field

Best practices

Do
Set a default height that comfortably fits the expected input length.
Don't
Don't set the initial height so small that the user can barely read what they're typing.
Do
Use a label to name the field and a placeholder for a short hint or example.
Don't
Don't use the placeholder as a substitute for the label — it disappears as soon as the user types.

Variants

The textarea has one shape. The label is optional and can be omitted when surrounding context provides a clear programmatic label.

Standard — a resizable multi-line input with an optional label above. Use whenever users need to enter more than a single line of text.
Without label — the textarea alone, without a visible label element. Only use this when a heading, table column header, or aria-label provides the accessible name in the surrounding UI.

Accessibility

→
Always pair <label for="…"> with <textarea id="…">. The matching for/id pair ensures screen readers announce the label when the textarea receives focus and expands the click target to the full label text.
→
Link helper text and error messages via aria-describedby. Give the .ta-help or .ta-err element an id and reference it on the <textarea>. Screen readers will announce it after the label.
→
Add aria-invalid="true" on error. Together with aria-describedby pointing to the error message, this gives assistive technology full context — the input is invalid and here's why.
→
Never set resize: none. Preventing resize can make the textarea unusable for people who need larger text or who paste long content. Only restrict resizing if the layout cannot accommodate variable height.
→
Don't rely on placeholder text as the only label. Placeholder disappears on input and has low contrast in most browsers, making it inaccessible to users with visual impairments.

Accessibility

→
Always pair <label for="…"> with <textarea id="…">. The matching for/id pair ensures screen readers announce the label when the textarea receives focus.
→
Use aria-describedby for helper text and errors. Give the .ta-help or .ta-err element an id and reference it with aria-describedby on the textarea.
→
Add aria-invalid="true" on error. Pair it with aria-describedby pointing to the error message so assistive technology announces both the invalid state and the reason.
