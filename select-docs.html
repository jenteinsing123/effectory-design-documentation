Select

A control that lets users choose one or more options from a list. Two variants: Single Select for picking one option and Multi Select for picking several. A name property renders the field with an inline semibold prefix instead of a stacked label.

Anatomy

Both variants share the same underlying <select> element wrapped in a stacked form-field layout. A name property is available to render the field with a semibold prefix label inline instead of stacked above — useful in toolbars and filter bars.

Usage

Use when
  • ✓ A user must pick from a predefined list too long to show as radio buttons or checkboxes (4+ options)
  • ✓ Multiple values are valid (Multi Select) — tags, categories, filters
  • ✓ Multiple independent filters sit side by side in a toolbar — set the name property to render them inline
  • ✓ The current value must always be visible without opening a panel first
Avoid when
  • ✗ Fewer than 4 options are available — use radio buttons or a segmented control instead
  • ✗ More than 3 named selects appear next to each other — consider a dedicated filter panel
  • ✗ The action triggers immediately rather than changing a view setting — use a button
  • ✗ A free-text answer is also valid — use a combobox or text field instead

Behaviour

→
Anchored to the trigger. The menu opens directly below the field by default, flipping above when there is not enough space.
→
Width. Menu matches trigger width. For named selects in toolbars, the menu may be wider than the trigger to fit longer option labels.
→
Search threshold. Show the search input when the list has more than 7 options or contains free-form values like countries, languages or users.
→
Closing. Single Select closes on selection. Multi Select stays open so users can toggle several values; it closes on outside click, Esc, or focus moving away.
→
Max height. Constrain to ~280px and scroll inside the menu rather than letting it grow off-screen.

Accessibility

→
Use a native <select> or a proper ARIA combobox. Native selects get keyboard and screen-reader support for free. Custom dropdowns must have role="combobox", aria-expanded, and aria-haspopup="listbox".
→
Always associate a visible label. Use <label for="..."> or aria-labelledby. Never rely on placeholder text as the label — it disappears on interaction.
→
Link validation messages with aria-describedby. Set the attribute on the field so screen readers announce the error or warning after reading the label.
→
Chip remove buttons need accessible names. aria-label="Remove Result" — the visible ✕ alone is not sufficient for screen readers.
→
Selects with the name property need an accessible name. The visible name prefix is decorative — mark it aria-hidden="true" and add aria-label="Sort" on the <select> so screen readers announce the full context, not just the current value.

With name (inline prefix)

Add the has-name modifier and a select-name span to render a semibold prefix inline before the value. Useful in toolbars and filter bars. Combine with multi-select for the multi variant.

Sort: Newest first 
<div class="select-field has-name">
  <span class="select-name" aria-hidden="true">Sort:</span>
  <select class="select" aria-label="Sort">
    <option value="newest">Newest first</option>
    <option value="oldest">Oldest first</option>
  </select>
</div>

Accessibility

→
Form fields need a <label>. Use <label for="id"> linked to the field's id. Never rely on placeholder text alone.
→
Error messages need aria-describedby. aria-describedby="field-error" on the field + id="field-error" on the message element.
→
Chip remove buttons need a label. <button aria-label="Remove Design"> — the ✕ glyph alone isn't descriptive enough.
→
Selects with name need aria-label. Add aria-label="Sort" directly on the <select> element. The visible name prefix is decorative — mark it aria-hidden="true" so it isn't read twice.
→
Custom dropdowns must manage focus. Move focus to the first option when the dropdown opens. Close on Escape and return focus to the trigger. Arrow keys navigate between options.
