Select

A family of controls that let users choose one or more options from a list. Three variants: Single Select for picking one option in a form, Multi Select for picking several, and Selection Button — a compact inline trigger for toolbars and filter bars.

Anatomy

Single Select and Multi Select share the same form-field shell. Selection Button is a standalone inline trigger with no label above it.

Variants

Three variants cover different contexts of use.

Single Select — pick exactly one option. Use inside forms, settings panels, and dialogs. Shows the current value once filled; chevron opens the dropdown.

Option
Multi Select — pick one or more options. Selected values appear as dismissible chips inside the field. The field grows vertically as chips accumulate. Use for tags, categories, or any filter where multiple answers are valid.
Selection Button — a compact inline trigger that always shows its active value. No label above. Use in toolbars and filter bars where space is tight and the current selection must stay visible. Comes in bordered and inline appearances.

Usage

Use when
  • ✓ A user must pick from a predefined list too long to show as radio buttons or checkboxes (4+ options)
  • ✓ Multiple values are valid (Multi Select) — tags, categories, filters
  • ✓ Multiple independent filters appear side by side in a toolbar (Selection Button)
  • ✓ The active selection should always stay visible without opening a panel
Avoid when
  • ✗ Fewer than 4 options are available — use radio buttons or a segmented control instead
  • ✗ More than 3 Selection Buttons appear in a row — consider a dedicated filter panel
  • ✗ The action triggers immediately rather than changing a view setting — use a button
  • ✗ A free-text answer is also valid — use a combobox or text field instead

Content guidelines

Good label copy reduces errors and makes forms faster to complete.

Don't start labels with a verb or write them as instructions. Don't replace "Select…" with custom placeholder text.

Keep the Selection Button name to one word ending with a colon — "Sort:", "Status:", "Filter:". Values should match the exact option label from the dropdown.
Don't
Don't use multi-word or verb names for the Selection Button. Don't truncate or rephrase the active value.

Selection Button

A compact inline trigger for toolbars and filter bars. Always shows the active value without opening a panel first. No disabled state — if the filter cannot be changed, hide the button or replace it with static text.

Appearances

Bordered — white background with a subtle border and shadow. Default appearance. Use in toolbars and filter bars on plain backgrounds.
Inline — no border or background at rest. Background appears only on hover and press. Use inside table headers and card sections where a visible border would add too much visual noise.

Best practices

Do
Summarise multiple active values as a count — "3 selected". Show "All" when nothing is filtered.
Don't
Don't list every selected value — the button overflows and becomes impossible to scan.

Accessibility

→
Use a native <select> or a proper ARIA combobox. Native selects get keyboard and screen-reader support for free. Custom dropdowns must have role="combobox", aria-expanded, and aria-haspopup="listbox".
→
Always associate a visible label. Use <label for="..."> or aria-labelledby. Never rely on placeholder text as the label — it disappears on interaction.
→
Link validation messages with aria-describedby. Set the attribute on the field so screen readers announce the error or warning after reading the label.
→
Chip remove buttons need accessible names. aria-label="Remove Result" — the visible ✕ alone is not sufficient for screen readers.
→
Selection Button needs aria-label. Since there is no form label above, use aria-label="Sort: Newest first" so the full meaning is announced, not just "button".

Accessibility

  

    
→
Form fields need a <label>. Use <label for="id"> linked to the field's id. Never rely on placeholder text alone.

    
→
Error messages need aria-describedby. aria-describedby="field-error" on the field + id="field-error" on the message element.

    
→
Chip remove buttons need a label. <button aria-label="Remove Design"> — the ✕ glyph alone isn't descriptive enough.

    
→
Selection Button needs aria-label. aria-label="Sort: Newest first" describes the full control, not just "button".

    
→
Custom dropdowns must manage focus. Move focus to the first option when the dropdown opens. Close on Escape and return focus to the trigger. Arrow keys navigate between options.
