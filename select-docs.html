Select

A control that lets users choose one or more options from a list. Two variants: Single Select for picking one option and Multi Select for picking several. A name property renders the field with an inline semibold prefix instead of a stacked label.

Anatomy

Both variants share the same underlying <select> element wrapped in a stacked form-field layout. A name property is available to render the field with a semibold prefix label inline instead of stacked above — useful in toolbars and filter bars.

Usage

Use when
  • ✓ A user must pick from a predefined list too long to show as radio buttons or checkboxes (4+ options)
  • ✓ Multiple values are valid (Multi Select) — tags, categories, filters
  • ✓ Multiple independent filters sit side by side in a toolbar — set the name property to render them inline
  • ✓ The current value must always be visible without opening a panel first
Avoid when
  • ✗ Fewer than 4 options are available — use radio buttons or a segmented control instead
  • ✗ More than 3 named selects appear next to each other — consider a dedicated filter panel
  • ✗ The action triggers immediately rather than changing a view setting — use a button
  • ✗ A free-text answer is also valid — use a combobox or text field instead

When to use search

→
More than ~7 options. Below that, scanning is faster than typing. Above that, scrolling becomes painful and search pays off.
→
Open-ended or unfamiliar values. Countries, languages, users, locations, projects — anything the user already knows by name. Set searchable on <eff-select>.
→
Async or remote data. When options come from a backend (people, organisations, surveys), use <eff-select> with disableLocalSearch and listen to searchTermChange.
→
Don't add search for short, fixed lists (sort orders, statuses, 3–5 categories). It only adds visual noise and a tab stop.

When to use a description

A description (subtitle) clarifies an option when the title alone is ambiguous, technical or otherwise insufficient. It is always optional — most lists do not need one.

→
Disambiguating similar titles. Two offices both called "Amsterdam" — show the address or team size as a subtitle.
→
Pairing a human label with a technical code. Locale codes (nl-NL), API keys, IDs, role slugs. Title carries meaning; subtitle carries the precise value.
→
Explaining a choice with consequences. Survey types, plan tiers, permission levels — short subtitle helps users understand what they'd actually be picking.
→
Adding context, not detail. Keep the subtitle to one line — it must truncate cleanly. Long-form explanations belong in a tooltip or in a separate help link.
→
Don't add a description when the title is already self-evident (countries, sort orders, simple categories). Empty subtitles add visual weight without value. To opt-in per option, set description on the EffectoryOption.

Behaviour

→
Anchored to the trigger. Opens directly below the field by default, flipping above when there is not enough space.
→
Width. Menu matches trigger width. For named selects in toolbars, the menu may be wider than the trigger to fit longer option labels.
→
Closing. Single Select closes on selection. Multi Select stays open so users can toggle several values; it closes on outside click, Esc, or focus moving away.
→
Max height. Constrain to ~280px and scroll inside the menu rather than letting it grow off-screen.
→
Select all / Deselect all. For multi-select lists, the action row at the bottom flips between "Select all" and "Deselect all" depending on current state. Filling the outline checkbox indicates everything visible is selected.

Accessibility

→
Use a native <select> or a proper ARIA combobox. Native selects get keyboard and screen-reader support for free. Custom dropdowns must have role="combobox", aria-expanded, and aria-haspopup="listbox".
→
Always associate a visible label. Use <label for="..."> or aria-labelledby. Never rely on placeholder text as the label — it disappears on interaction.
→
Link validation messages with aria-describedby. Set the attribute on the field so screen readers announce the error or warning after reading the label.
→
Chip remove buttons need accessible names. aria-label="Remove Result" — the visible ✕ alone is not sufficient for screen readers.
→
Selects with the name property need an accessible name. The visible name prefix is decorative — mark it aria-hidden="true" and add aria-label="Sort" on the <select> so screen readers announce the full context, not just the current value.

Two implementations are available. <mat-select> (Angular Material) covers the basic stacked, inline and multi cases. For richer needs (search, validation, large lists, locale picking) use the Effectory wrappers <eff-select>, <eff-select-virtual>, and <eff-select-language>.

When to use which

COMPONENTUSE WHEN
<mat-select>Short, static option lists. No search needed. Standard Material API.
<eff-select>You need search, validation labels, or programmatic options[].
<eff-select-virtual>Hundreds or thousands of options. Same API as eff-select, with virtual scrolling.
<eff-select-language>Locale picking. Renders flag glyph + auto-translates option text.
<mat-autocomplete>Free-text input where users can also pick a suggested value.

Accessibility

→
Wrap the field in <mat-form-field> with a <mat-label>. Material wires the label to the field automatically — never rely on placeholder as the only descriptor.
→
Standalone triggers need aria-label. When you don't render a visible label (e.g. inline class="inline" selects in a toolbar), add aria-label="Sort" directly on the <mat-select>.
→
Validation errors need <mat-error>. Place inside the <mat-form-field>. Material handles aria-describedby wiring. For <eff-select>, use invalidValueError for invalid-option messaging.
→
Icon-only triggers need a label. <eff-mat-icon> is decorative by default. If the icon carries meaning, set aria-label on the option, not on the icon.
→
Don't reach for disableRipple for a11y reasons. It's a visual choice — Material's keyboard handling (arrow keys, Enter/Space, Escape, type-ahead) and focus management are unaffected.
→
Async lists must announce loading. When using [isLoading]="true" with disableLocalSearch, ensure the loading region is announced — Material's listbox role + the live status text covers most cases, but verify with a screen reader.
