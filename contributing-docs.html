Contributing

How to propose changes, report issues, and help grow the design system.

Ways to contribute

The design system is a shared resource — anyone on the product team can contribute to it.

The Engage Design System is not maintained exclusively by the DS team. It improves when the people using it daily — designers, developers, and PMs — flag problems and bring ideas. All three routes go through Jira. The DS team triages new tickets weekly and brings larger proposals to the bi-weekly sync.

How to contribute

Reporting a bug

A clear bug report saves significant back-and-forth. Include enough context to reproduce it without asking follow-up questions.

Create a Jira ticket with the label design-system and include the following:

Which component — name the component (e.g. eff-select) and, if relevant, the Angular selector or Figma component path.
Which state or variant — be specific: disabled state, focus ring on hover, the "bordered" checkbox variant, etc.
A screenshot or recording — annotate it if the issue is subtle. For code bugs, a Storybook link or a minimal reproduction is ideal.
Where the issue lives — Figma, code (Angular component), the documentation site, or a combination. This determines who picks it up first.

Proposing a new component

New components are added deliberately — each one carries a long-term maintenance cost.

Use the "Component proposal" Jira template and fill in the following fields:

Field What to include
Use case What problem does this component solve? Why is an existing component not sufficient?
Existing usage Where is this pattern already appearing in the product? Link to screens or Jira tickets.
System alignment Which existing components should it visually and behaviourally align with?
Figma sketch A rough layout, even hand-drawn, helps the conversation move faster. Not required but strongly encouraged.

The DS team reviews all proposals in the bi-weekly sync. You are welcome to join that meeting to walk through your proposal in person — comment on the ticket to request an invite.

Design contributions (Figma)

All Figma contributions work through branching off the main library file.

The main Figma library file is the source of truth for all design specifications. To propose a change:

1
Branch off the main library
Create a branch off the main library file. Name it after your Jira ticket (e.g. ENG-1234-checkbox-indeterminate).
2
Follow naming conventions
Follow the existing naming conventions for all layers and components: Component / Variant / State (e.g. Checkbox / Bordered / Indeterminate).
3
Use only library variables
Use only library variables for every colour and spacing value. Do not introduce raw hex values or hardcoded numbers — if a token doesn't exist for what you need, that is a separate proposal.
4
Request a DS review
Request a review from the DS team by tagging them in a Figma comment on the branch. At least one DS team member must approve before a branch is merged.

Documentation updates (usage guidelines, anatomy annotations) in Figma follow the same branching flow.

Development contributions

Follow the existing API conventions and add Storybook coverage before opening a PR.

1
Fork and branch
Fork the component library repository and branch off main. Branch names should follow feat/eff-component-name or fix/eff-component-name-issue.
2
Follow API conventions
Follow the existing component API conventions: Angular [input] bindings use descriptive noun names ([disabled], [size]); (output) events use past-tense verbs ((valueChange), (closed)).
3
Style via CSS custom properties only
No raw colour values or hardcoded pixel values in component stylesheets. Every visual property must map to a token.
4
Add Storybook stories
Add or update Storybook stories. Each variant and each interactive state should have its own story. Stories serve as the live API documentation.
5
Open a PR
Open a PR against main and tag it with the design-system label. Link to the corresponding Jira ticket and, if applicable, the Figma branch that specifies the design.

PRs require at least one approval from the DS team and passing CI (lint, unit tests, visual regression) before merging.

Versioning and deprecation

The system uses semantic versioning. Breaking changes are signalled well in advance.

The component library follows semver. In practice this means:

  • Patch releases (e.g. 1.4.1) — bug fixes that are safe to adopt immediately.
  • Minor releases (e.g. 1.5.0) — new components or non-breaking additions. Also where deprecations are announced.
  • Major releases (e.g. 2.0.0) — breaking API changes. Rare and communicated well ahead of time.

When an API or component is deprecated, it is flagged with @deprecated in JSDoc with a note on the recommended replacement. The deprecation notice appears in the changelog at least one sprint before the feature is removed. Deprecated Figma components are marked with a "Deprecated" badge in the library.

If you encounter a deprecated API that has no clear migration path, raise a Jira ticket rather than continuing to use it — it may indicate a gap in the replacement component.
