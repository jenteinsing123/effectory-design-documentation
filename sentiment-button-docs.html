Sentiment Button

A button coloured by sentiment — positive, negative, informative or warning. Use it for an action that carries the meaning of that sentiment, like confirming a positive step or a destructive one.

Anatomy

A standard button with a sentiment colour applied to its border (secondary) or fill (primary).

1
Label — Names the action; 14px Medium.
2
Sentiment colour — Border (secondary) or fill (primary), set by the sentiment: positive, negative, informative, warning.
3
Icon — Optional leading or trailing 16px icon.

Usage

Use a sentiment button when the action itself carries that sentiment — "Approve", "Delete", "Dismiss warning". For ordinary actions use the regular Button; sentiment colour loses meaning if everything is coloured.

Reserve colour for meaning

Do
One sentiment action (the destructive one), paired with a neutral.
Don't
Colour every action — the sentiment stops meaning anything.

Variants

Two properties: the sentiment (the colour and meaning) and the emphasis (secondary outline or primary fill).

Sentiment

Positive — A constructive or confirming action: approve, save, publish.
Negative — A destructive or risky action: delete, remove, revoke.
Informative — A neutral-but-noteworthy action tied to an info message.
Warning — An action that needs caution but isn't destructive.

Emphasis

Secondary — White fill with a coloured border. The default; sits quietly next to neutral actions.
Primary — Filled with the sentiment colour for the single highest-emphasis action in a flow.

Content guidelines

Lead with the verb; the colour already sets the tone.

Name the action, don't repeat the sentiment

The colour conveys positive/negative — the label should say what happens.

Do
A clear verb + object.
Don't
Restating the sentiment in the label.

Design Tokens

Each sentiment maps to its border/fill token set; text is base (or white on a dark fill).

SentimentSecondary borderPrimary fill
Positive--border-positive-base--bg-positive-base
Negative--border-negative-base--bg-negative-base
Informative--border-info-base--bg-info-base
Warning--border-warning-base--bg-warning-bold (dark text)

Hover uses the matching -base-hover token; secondary pressed uses -subtle; text --content-base (or --content-on-brand-base on filled info/positive/negative). Shadow --sh-action → --sh-action-hover. Height 36, radius --radius-base, label body-14 Medium.

Accessibility

Colour adds meaning but can't be the only signal.

→
Label carries the meaning. Don't rely on colour alone — the text must say what the action does (a red button must still read "Delete").
→
Contrast. Filled info/positive/negative use white text; the light warning fill keeps dark text for contrast.
→
Disabled. A disabled button is non-focusable and not announced as actionable.
→
One per group. Limit to one sentiment action per group so the emphasis stays clear.

Usage

Build the button from the .sbtn base plus a sentiment modifier; add .is-primary for the filled emphasis.

Sentiments

<button class="sbtn is-positive">Positive</button>
<button class="sbtn is-negative">Negative</button>
<button class="sbtn is-info">Informative</button>
<button class="sbtn is-warning">Warning</button>

Primary (filled)

<button class="sbtn is-primary is-negative">Delete</button>
⚠️
Figma gap — Angular API Figma documents the sentiment button as a standalone control with both a secondary and a filled primary variant. In the Angular styleguide it currently only appears as the secondary sentiment action inside an Inline notification (via buttonMessages / messageType) — there's no standalone filled-primary sentiment button yet. Confirm the intended Angular API before building.
⚠️
Figma gap — primitive tokens In Figma some states are painted with primitive colour tokens (e.g. color/orange/700, color/blue/900, color/green/900 for the primary pressed states). Primitives shouldn't be referenced directly. This documentation uses the semantic tokens instead (--bg-{sentiment}-base/-hover/-pressed, --border-{sentiment}-base/-hover); the Figma layers should be rebound to those semantic tokens.

Accessibility

A native button with a meaningful label.

→
Use a <button>. Real button semantics, keyboard-operable.
→
Label, not colour. The text must carry the action's meaning on its own.
→
Disabled. Use the disabled attribute so it's removed from the tab order.
