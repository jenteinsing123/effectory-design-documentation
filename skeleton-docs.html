Skeleton

Placeholder shapes that mimic the layout of content while it loads. They set expectations and make a screen feel faster than a spinner when you know what's coming.

Anatomy

Simple shapes — lines, circles and blocks — arranged in the same layout as the content they stand in for, with a soft pulse.

1
Line — A placeholder for a line of text; the last line is usually shorter.
2
Circle — Stands in for an avatar or round icon.
3
Block — A larger area for an image, thumbnail or chart.
4
Pulse — A soft pulse between two greys that signals the screen is loading, not frozen.

Usage

Use a skeleton when you know the layout that's coming — lists, cards, tables, a profile header. For a short or unpredictable wait, a Loading spinner is simpler; for measurable progress, a Progress Bar.

Mirror the real layout

Do
Shapes that match the headings and text that will load.
Don't
A lone shape that doesn't reflect the real layout — use a spinner instead.

Variants

Compose a skeleton from a few primitive shapes; size them to match the content.

Line — A text line (12px tall). Vary the width per line; make the last one shorter for a natural look.
Title — A taller line (20px) for headings.
Circle — For an avatar or round icon; match the avatar size (e.g. 32 / 40px).
Block — A rectangle for images, thumbnails or charts; set its width and height.

States

A skeleton has one state — loading — and is then replaced by the real content.

Loading
Loaded
The real content takes the same space the skeleton held.

Design Tokens

A neutral placeholder fill with a lighter shimmer sweep.

TokenRoleValue
--border-basePlaceholder fill (rest)#e1e4e8
--bg-base-pressedPulse highlight#f1f2f4
--radius-smText line radius4px
--radius-mdBlock radius8px
--radius-fullCircle999px

Pulse: the fill eases between --border-base and --bg-base-pressed every ~1s — matching the styleguide's table loader. Honours prefers-reduced-motion (static fill).

Accessibility

Skeletons are decorative — the loading state is what needs to be announced.

→
Hide the shapes. Mark skeletons aria-hidden="true" so screen readers don't read empty placeholders.
→
Announce loading. Set aria-busy="true" on the region and expose a status (e.g. a visually-hidden "Loading") so the wait is perceivable.
→
Swap, don't shift. The skeleton should occupy the same space as the real content to avoid layout jumps when it loads.
→
Respect reduced motion. The shimmer stops under prefers-reduced-motion; the static placeholder still communicates loading.

Usage

Compose skeletons from the .skeleton base plus a shape modifier; size each shape with inline width/height to match the real content.

Shapes

<div class="skeleton skeleton-text" style="width:200px"></div>
<div class="skeleton skeleton-title" style="width:160px"></div>
<div class="skeleton skeleton-circle" style="width:40px;height:40px"></div>
<div class="skeleton skeleton-block" style="width:220px;height:80px"></div>

Card skeleton

<div aria-busy="true" aria-label="Loading">
  <div class="skeleton skeleton-circle" style="width:40px;height:40px"></div>
  <div class="skeleton skeleton-text" style="width:80%"></div>
  <div class="skeleton skeleton-text" style="width:60%"></div>
</div>

Table loading

In a mat-table, the styleguide fills each cell with a pulsing .loading-bar (full-width, ~12px tall, 8px radius) — the same pulse as the skeleton above.

<!-- one per cell while the row loads -->
<td><span class="loading-bar"></span></td>
⚠️
Confirm the Angular API The styleguide uses a pulsing .loading-bar for table loading, but has no general skeleton component for arbitrary layouts (cards, headers, lists). The .skeleton classes here are the design-system pattern; align on a shared .loading-bar / eff-skeleton approach with engineering before building.

Accessibility

Hide the shapes; announce the loading state.

→
aria-hidden. Put aria-hidden="true" on skeleton shapes.
→
aria-busy. Set aria-busy="true" on the loading region; remove it (and the skeletons) when content arrives.
→
No layout shift. Match the skeleton's size to the content it replaces.
