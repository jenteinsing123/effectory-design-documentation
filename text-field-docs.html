Text Field

A single-line input for free-form text entry — from plain names and emails to icon-decorated search fields and compact inline cell editors.

Anatomy

The text field stacks a label, an input container, and optional supporting text — with icons optionally slotted inside the container.

1
Label — a short, descriptive text above the field. Always required unless the field is in a context that provides an unambiguous label (e.g. an inline table cell). Use sentence case.
2
Container — the bordered rectangle that holds the input. Carries the background, border, corner radius, and shadow. Colour changes communicate state (hover, focus, error, disabled).
3
Placeholder / value — muted placeholder text disappears as soon as the user starts typing. Once filled, the value text is full contrast. Never use placeholder text as a substitute for a label.
4
Icon (optional) — a 16×16 icon slotted inside the container on the leading or trailing edge. Use leading icons to signal input type (search, email, lock); use trailing icons for actions (clear, reveal password).
5
Helper text (optional) — a short line below the field that provides format hints or context. Replaced by an error message when the field enters an invalid state.

Usage

Use a Text Field whenever a user needs to enter a short, free-form value that doesn't fit a constrained control like a Select or Toggle.

Use when
  • The answer is open-ended and can't be enumerated — names, emails, search queries, free text
  • A single line is enough; use Textarea for multi-line answers
  • Input needs contextual affordance — a search icon, password reveal, or unit suffix
  • Editing a value inline inside a table row or settings list (use the Inline variant)
Avoid when
  • The set of valid answers is finite — use Select or Radio Group instead
  • The answer requires structured formatting with keyboard control — use Date Picker for dates
  • The question is binary — use a Checkbox or Toggle instead

Accessibility

→
Always use a native <input> with an associated <label>. Connect them with matching for and id attributes. This gives screen readers the correct announced name and expands the clickable hit area to the whole label text.
→
Never rely on placeholder text as the only label. Placeholder text has a contrast ratio below WCAG AA, disappears when the user types, and is not reliably announced by all screen readers.
→
Mark required fields with aria-required="true". Also include a visible indicator (the asterisk *) and explain its meaning once at the top of the form — e.g. "Fields marked with * are required".
→
Link helper text and error messages via aria-describedby. Give the .tf-help or .tf-err element an id and reference it on the <input> with aria-describedby. Screen readers will then announce the helper/error after the field label.
→
Set aria-invalid="true" on fields in an error state. This signals to assistive technology that the input contains a problem, independent of the visual red border. Combine with a descriptive error message via aria-describedby.
→
Use the most specific type attribute. type="email", type="tel", and type="number" trigger the appropriate mobile keyboard, enable browser autofill, and help screen readers announce the expected format.

Accessibility

→
Always pair <label for="…"> with <input id="…">. The matching for/id pair ensures screen readers announce the label when the input receives focus and expands the click target to the full label text.
→
Use aria-describedby for helper text and errors. Give the .tf-help or .tf-err element an id and reference it with aria-describedby on the input. Screen readers will announce it after the label and value.
→
Add aria-invalid="true" on error. Together with aria-describedby pointing to the error message, this gives assistive technology full context — the input is invalid and here's why.
→
Use aria-required="true" on required fields. This is separate from the HTML required attribute — use both: required triggers browser validation; aria-required signals to screen readers that the field must be filled before submission.
→
Mark decorative icons with aria-hidden="true". Icons inside .tf-field-icon are cosmetic — they must be hidden from the accessibility tree so screen readers don't read out meaningless SVG content.
→
The inline variant needs a programmatic label. Since .tf-inline has no visible <label>, add an aria-label attribute or use aria-labelledby pointing to a contextual heading in the surrounding UI (e.g., a table column header).
