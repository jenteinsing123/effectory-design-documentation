Inline notification

A persistent, non-blocking message that sits inline with page content to surface status, warnings, or contextual announcements.

Anatomy

An inline notification combines a status icon, a clear message, optional actions, and a way to dismiss.

Heads up! This alert needs your attention
1
Icon badge — A coloured circular icon that signals the message type (info, warning, positive, negative) at a glance.
2
Title — A short, scannable label that summarises the message in one or two words.
3
Message — The body text explaining what happened or what the user should do.
4
Close button — Lets the user dismiss the notification. Optional but recommended unless the message is critical.
+
Actions (optional) — A link or one or more buttons placed before the close, used when the notification offers a direct follow-up action.

Usage

Use inline notifications when a message is tied to the surrounding content and the user should still see and interact with the page.

Choose the right type

Well done!Your changes have been saved
Do
Match the type to the situation — Positive for success, Negative for errors, Warning for things needing attention, Info for neutral context.
Oh snap!Your changes have been saved
Don't
Pair a Negative variant with a success message. The colour and tone send conflicting signals and erode trust in the system.

Keep messages concise

Warning!Your subscription expires in 3 days
Do
Lead with the most important information. Title states what; message says what to do or expect next.
Warning!We've noticed that your subscription is scheduled to expire in three days from now and you may want to take action soon to avoid losing access
Don't
Pack the message with filler. Long-form prose belongs in a dialog or a dedicated page, not in an inline notification.

Variants

Five message types cover the most common signals; two layouts handle short and longer copy.

By type

New feature!We released a new feature you might want to use
Feature — Announce a new feature, capability, or product update. Pair with a link or button that lets the user explore the feature directly. Use sparingly; reserve for genuinely new things.
Heads up!This alert needs your attention, but it is not super important
Info — Neutral, contextual information the user should know but doesn't need to act on immediately. Examples: "Read-only mode active", "3 invitations pending".
Warning!Better check yourself, you're not looking too good
Warning — Something needs the user's attention soon to avoid a problem. Use for expiring trials, deprecations, near-quota limits, or actions that may have unintended side effects.
Well done!You successfully read this important alert message
Positive — Confirms a successful outcome that persists on the page (saved settings, imported data, completed setup). For transient success, prefer a Toast.
Oh snap!Change a few things up and try again
Negative — Errors or failures the user must address before they can continue. Examples: failed import, permission denied, API error.

By layout

Heads up!This alert needs your attention
Compact — Title and message flow on a single line. Use for short messages that fit without wrapping. Default layout.
Heads up!This alert needs your attention, but it is not super important
Stacked — Title above the message. Use when the message is longer than ~5 words or when actions are present. Add the is-stacked modifier.

Content guidelines

Notifications are read at a glance. Keep titles short, messages specific, and actions verb-led.

  • Title (1–3 words) — Use a short label that mirrors the type. Well done!, Warning, Heads up, New feature, Couldn't save.
  • Message (one sentence) — State what happened and, if relevant, what the user should do next. Avoid jargon and apologies.
  • Action label — Start with a verb that describes the outcome: Review, Try again, Learn more. Never Click here.
  • Punctuation — End full sentences with a period. Single-word titles can use an exclamation only when it adds meaning.

Accessibility

Inline notifications must be perceivable, operable, and announce themselves to assistive tech.

→
Pick the right role. Use role="status" for Info, Positive, and Feature notifications and role="alert" for Warning and Negative. The role determines how screen readers announce it.
→
Never colour alone. The variant colour is paired with an icon and a textual title; never rely on colour alone to communicate the message type.
→
Label the close. Always include an aria-label like Dismiss notification. Focus must be visible — the :focus-visible ring is built in.
→
Keyboard reachable. The close button is reachable via Tab and activated with Enter or Space.
→
Contrast. All title and message text meets WCAG 2.2 AA (≥4.5:1) on the variant background.
→
Don't trap focus. Inline notifications are non-modal. Never move focus to them on mount unless the user just triggered an error they need to fix.

Usage

Drop a notification anywhere in the document flow. Pair the right variant class with the matching icon.

Info — compact

Heads up!This alert needs your attention
<div class="inline-notif is-info">
  <span class="inline-notif-icon is-info"><i data-icon="info"></i></span>
  <div class="inline-notif-content">
    <span class="inline-notif-title">Heads up!</span>
    <span class="inline-notif-msg">This alert needs your attention</span>
  </div>
  <button class="inline-notif-close" aria-label="Dismiss"><i data-icon="cross"></i></button>
</div>

Accessibility

Pick the right role, label the close, and let users dismiss via keyboard.

→
Match the role to the type. Wrap Warning and Negative variants in role="alert"; wrap Info, Positive, and Feature in role="status".
→
Label the close. The close button must have a descriptive aria-label (e.g. aria-label="Dismiss notification").
→
Don't steal focus. Keep focus order natural — don't move focus to the notification on mount unless it's a blocking error.
→
Live regions outside the dismiss button. If the message changes dynamically, place the live region outside the close button so screen readers don't re-announce on dismiss.
