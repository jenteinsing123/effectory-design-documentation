Buttons

Buttons are clickable elements used to trigger actions. They communicate calls to action and allow users to interact with pages. Each page should have only one primary button — all remaining actions should use lower-emphasis variants.

States

Default

Resting state. No user interaction. The canonical appearance of the button.

Hover

Cursor is over the button. Provides visual feedback that the element is interactive.

Pressed

Button is actively being clicked or tapped. Confirms the interaction is registered.

Disabled

Action is unavailable. Non-interactive and excluded from keyboard tab order.

⚠️ filter-button has no Disabled state by design — remove a filter from the UI rather than disabling it.

Type descriptions

TypeWhen to useExample labels
Primary Highest emphasis. One per page/section. The single most important action. Save, Submit, Confirm, Create
Secondary Medium emphasis. Supporting actions that are important but not the main CTA. Cancel, Edit, Export
Tertiary Low emphasis. Optional, supplementary actions where visual weight should be minimal. Learn more, View details, Skip
Danger Destructive actions that are irreversible or have significant consequences. Delete account, Remove, Revoke access

Accessibility
⌨️

Keyboard interaction

  • Reachable via Tab, activated via Enter or Space.
  • Disabled buttons must be excluded from tab order — use the native disabled attribute.
  • Do not use aria-disabled="true" as a sole substitute unless keeping buttons focusable for discoverability.
🏷️

Labels & accessible names

  • button, button-big, link-button, sentiment-button — visible text is the accessible name.
  • icon-button — no visible label. Always add aria-label describing the action, not the icon (e.g. "Close dialog", not "X").
  • filter-button — accessible name should include both filter name and value as a single string.
🔲

Focus indicators

  • All focusable buttons must display a visible focus ring.
  • Never remove :focus-visible outline without a custom equivalent.
  • Focus rings must achieve 3:1 contrast ratio against adjacent colours (WCAG 2.2 §2.4.11).
🎨

Colour & contrast

  • Text and icon colours must meet 4.5:1 contrast against background in all interactive states.
  • Disabled state is WCAG-exempt, but aim for 3:1 minimum to aid low-vision users.
  • Danger and Negative types must not rely on colour alone — the label must communicate the destructive intent.
  • Always verify Inverse link-button contrast in context before use.
🏗️

Semantics

  • Use native <button> wherever possible. Avoid div role="button".
  • Use <a href> for link-buttons that navigate. Use <button> for in-page actions.
  • Sentiment type must be reflected in the visible label or aria-label, not just colour.
🌀

Motion

  • If hover/pressed state transitions involve animation, respect prefers-reduced-motion.
  • Reduce or eliminate transition durations when motion is restricted.
Usage Guidelines
✅ Do

Use one Primary button paired with a lower-emphasis Secondary or Tertiary for supporting actions.

❌ Don't

Never place two Primary buttons side by side. If both seem equally important, reconsider the IA.

✅ Do

Use a Danger button with a confirmation dialog for irreversible actions. The label must describe the consequence.

❌ Don't

Avoid vague Danger button labels. "OK" or "Submit" on a red button gives no indication of what will be destroyed.

✅ Do

Set Primary=True on the recommended sentiment action and Primary=False on the alternative.

❌ Don't

Never use two filled (Primary=True) sentiment buttons in the same interaction group.

Button hierarchy

Highest
One per page. The single most important CTA.
Medium
Supporting actions alongside a primary.
Low
Optional or supplementary actions. Minimal visual footprint.
Caution
Irreversible or destructive actions only. Always pair with confirmation.
