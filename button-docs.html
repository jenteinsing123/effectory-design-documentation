Button

A labelled button that triggers an action. Use it whenever an action needs to be clearly named — it scales from low-emphasis ghost actions to high-emphasis primary calls-to-action and destructive confirmations.

Anatomy

The button is built from three layers. The label is always required; icons are optional.

① Container
② Icon (optional)
③ Label
1
Container — the outer shape that carries the fill, border, corner radius, and shadow. All visual differences between types and states are applied here.
2
Icon (optional) — a 14×14px icon placed left or right of the label. Use it to reinforce the action, not to decorate.
3
Label — always required. Keep it short and action-oriented: "Save", "Delete project", "Send invite". Avoid vague labels like "OK" or "Click here".

Usage

A button is the right choice when the action needs a clear label. Use type and hierarchy to help users understand what happens next.

✔ Use when
  • ✓The action has a clear name that fits in 1–3 words
  • ✓You need to communicate action hierarchy — one primary, supporting secondaries
  • ✓The action is destructive and requires an explicit, labelled confirmation
✕ Avoid when
  • ✕Space is too tight for a label — use Icon Button instead
  • ✕The action navigates to a new page — use a Link Button
  • ✕You need more than two primary buttons side by side — reconsider the hierarchy

Best practices

Do
Use one Primary button per view. Pair it with a Secondary for the opposing action.
Don't
Avoid two Primary buttons side by side — they create competing calls-to-action with no clear winner.
Do
Place the Primary action on the right. This matches the natural reading direction toward confirmation.
Don't
Don't place the Primary button on the left — users may reach for Cancel by habit and miss it.
Do
Give destructive actions visual breathing room — move them to the opposite side of the layout.
Don't
Never place a Danger button directly next to a Primary — users can accidentally click the wrong one.
Survey name is required
Do
Keep the button active and show an inline validation error so users know exactly what's blocking them.
Don't
Avoid disabled buttons without explanation — users get no feedback on why they can't proceed.

Alignment and positioning

Do
Right-align button groups in forms and dialogs — this anchors the primary action near the natural endpoint of the reading flow.
Don't
Avoid center-aligning buttons in forms — it disconnects them from the content edge and creates awkward visual tension.
Do
In page headers and toolbars, place action buttons on the trailing edge (right) so they don't compete with the page title.
Don't
Don't scatter buttons across the header — placing them before the title makes the layout feel unstructured and breaks scan order.

Variants

The Type property controls visual hierarchy. Use one Primary per view maximum.

Types
TYPEUSE FOR
Primary The single highest-emphasis action on a view. Submit a form, complete an onboarding step, confirm a key decision. Use only once per page section.
Secondary Supporting actions alongside a Primary — Cancel, Edit, Export. Can appear multiple times without competing with the primary.
Tertiary Low-emphasis inline actions in dense layouts — table rows, cards, toolbars. No background or border so it blends into content.
Danger Destructive, irreversible actions: delete, remove, revoke. Always requires an explicit label so users understand the consequence.

Content guidelines

Button labels are the voice of an action. Write them to be clear, specific, and self-explanatory — even when read out of context by a screen reader.

Use verb + noun

Combine an action verb with the object it acts on. This makes the outcome clear without relying on surrounding context.

Do
Verb + noun makes the outcome crystal clear, even without visual context.
Don't
Vague labels like "Submit", "Yes", or "OK" lose all meaning when read out of context.

Keep labels short

Aim for 1–3 words. If you feel the need for more, consider whether the extra context belongs in nearby helper text instead.

Do
Short labels scan instantly and fit neatly in the button container.
Don't
Long labels are hard to scan and can break layouts on smaller screens.

Icons reinforce, not decorate

Only add an icon when it genuinely aids comprehension — a trash icon on Delete, a plus on Add. When in doubt, leave the icon out.

Do
The trash and plus icons directly reinforce the meaning of their labels.
Don't
Random icons add visual noise without helping users understand the action faster.

Accessibility

→
Always use a native <button> element. You get keyboard focus, Enter/Space activation, and role="button" for free. Never simulate a button with a <div> or <span>.
→
Labels must describe the action, not the element. Write "Delete survey" not "Delete" when context is ambiguous. Screen readers read the label in isolation.
→
Disabled buttons need a reason. Add a tooltip or helper text explaining why the button is disabled. disabled buttons are not reachable by keyboard — consider using aria-disabled="true" with a click handler that explains the constraint instead.
→
Danger buttons must be extra clear. Use an explicit label ("Delete account", not just "Delete") and consider a confirmation dialog for irreversible actions.
→
Icon-only decoration needs aria-hidden. If an icon is next to a label, add aria-hidden="true" to the SVG so screen readers don't read it twice.

Installation

The Button component is part of @effectory/engage-ui. Install the package once and import per component.

npm npm install @effectory/engage-ui 
import { EfButton } from '@effectory/engage-ui'
import '@effectory/engage-ui/dist/style.css' // once per app entry

Usage

Basic

<EfButton variant="primary" @click="handleSave">Save changes</EfButton>

All variants

<!-- Primary: highest-emphasis action -->
<EfButton variant="primary">Save changes</EfButton>

<!-- Secondary: supporting action -->
<EfButton variant="secondary">Cancel</EfButton>

<!-- Tertiary: low-emphasis inline action -->
<EfButton variant="tertiary">Edit</EfButton>

<!-- Danger: destructive / irreversible actions -->
<EfButton variant="danger">Delete survey</EfButton>

Sizes

<EfButton size="sm">Small</EfButton>
<EfButton size="md">Medium (default)</EfButton>
<EfButton size="lg">Large</EfButton>

With icons

<!-- Icon on the left (leading) -->
<EfButton variant="primary" icon-left="plus">Add member</EfButton>

<!-- Icon on the right (trailing) -->
<EfButton variant="secondary" icon-right="download">Export</EfButton>

Loading & disabled states

<!-- Shows a spinner, button stays clickable-width -->
<EfButton variant="primary" :loading="isSaving">Save changes</EfButton>

<!-- Disabled — prefer inline validation instead (see Design tab) -->
<EfButton variant="primary" :disabled="!isFormValid">Save changes</EfButton>

<!-- aria-disabled keeps keyboard focus while signalling blocked state -->
<EfButton variant="primary" :aria-disabled="!isFormValid" @click="showErrors">Save changes</EfButton>

As a link

<!-- Renders an <a> tag instead of <button> -->
<EfButton variant="secondary" href="/reports">View report</EfButton>

<!-- Or with Vue Router -->
<EfButton variant="secondary" :to="{ name: 'reports' }">View report</EfButton>

Props

PROP TYPE DEFAULT DESCRIPTION
variant 'primary' | 'secondary' | 'tertiary' | 'danger' 'primary' Visual style and emphasis level of the button.
size 'sm' | 'md' | 'lg' 'md' Controls height and horizontal padding. sm = 28px, md = 36px, lg = 44px.
type 'button' | 'submit' | 'reset' 'button' Native HTML button type. Set to 'submit' inside forms.
disabled boolean false Disables the button. Removes from tab order. Prefer aria-disabled + validation error when possible.
loading boolean false Replaces label with a spinner. Button stays the same width to avoid layout shift.
icon-left string — Name of an Effectory icon to render on the leading side of the label.
icon-right string — Name of an Effectory icon to render on the trailing side of the label.
href string — When set, renders an <a> element instead of <button>.
to string | RouteLocationRaw — Vue Router destination. Renders a <RouterLink> when provided.

Accessibility

→
Use type="submit" inside forms. The default type="button" prevents accidental form submission, but you must explicitly set type="submit" on your submit button.
→
Prefer aria-disabled over disabled. The native disabled attribute removes the button from the tab order. Use aria-disabled="true" with a click handler that shows validation errors so keyboard users still get feedback.
→
Add aria-label when the label alone is ambiguous. For repeated actions in a table row (e.g. "Delete"), add aria-label="Delete survey: Q4 Engagement" to give screen reader users context.
→
Mark decorative icons with aria-hidden. Icons rendered alongside a visible label should have aria-hidden="true" so screen readers don't announce them twice.
