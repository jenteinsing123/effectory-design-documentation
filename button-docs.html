Button

A labelled button that triggers an action. Use it whenever an action needs to be clearly named — it scales from low-emphasis ghost actions to high-emphasis primary calls-to-action and destructive confirmations.

🔬 Anatomy

The button is built from three layers. The label is always required; icons are optional.

① Container
② Icon (optional)
③ Label
1
Container — the outer shape that carries the fill, border, corner radius, and shadow. All visual differences between types and states are applied here.
2
Icon (optional) — a 14×14px icon placed left or right of the label. Use it to reinforce the action, not to decorate.
3
Label — always required. Keep it short and action-oriented: "Save", "Delete project", "Send invite". Avoid vague labels like "OK" or "Click here".

✅ When to Use

✔ USE WHEN
  • ✓ The action has a clear name that fits in 1–3 words
  • ✓ You need to communicate action hierarchy — one primary, supporting secondaries
  • ✓ The action is destructive and needs an explicit, labelled confirmation (Danger type)
✕ AVOID WHEN
  • ✗ Space is too tight for a label — use an Icon Button instead
  • ✗ The action navigates to a new page — use a Link Button
  • ✗ You need more than two primary buttons side by side — reconsider the hierarchy

🎨 Variants

The Type property controls visual hierarchy. Use one Primary per view maximum.

Types
TYPEUSE FOR
Primary The single highest-emphasis action on a view. Submit a form, complete an onboarding step, confirm a key decision. Use only once per page section.
Secondary Supporting actions alongside a Primary — Cancel, Edit, Export. Can appear multiple times without competing with the primary.
Tertiary Low-emphasis inline actions in dense layouts — table rows, cards, toolbars. No background or border so it blends into content.
Danger Destructive, irreversible actions: delete, remove, revoke. Always requires an explicit label so users understand the consequence.

♿ Accessibility

→
Always use a native <button> element. You get keyboard focus, Enter/Space activation, and role="button" for free. Never simulate a button with a <div> or <span>.
→
Labels must describe the action, not the element. Write "Delete survey" not "Delete" when context is ambiguous. Screen readers read the label in isolation.
→
Disabled buttons need a reason. Add a tooltip or helper text explaining why the button is disabled. disabled buttons are not reachable by keyboard — consider using aria-disabled="true" with a click handler that explains the constraint instead.
→
Danger buttons must be extra clear. Use an explicit label ("Delete account", not just "Delete") and consider a confirmation dialog for irreversible actions.
→
Icon-only decoration needs aria-hidden. If an icon is next to a label, add aria-hidden="true" to the SVG so screen readers don't read it twice.
