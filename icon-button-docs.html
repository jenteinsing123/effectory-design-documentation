Icon Button

A fixed-size button that uses an icon instead of a text label to represent an action. Use it when space is tight and the icon's meaning is immediately clear to your audience.

Anatomy

The component has two parts. Every variant contains both layers.

1Container
2Icon
1
Container — the outer shape that defines size, background colour, border, corner radius, and shadow. All visual changes between types and states happen here.
2
Icon — the symbol inside the button. Any icon from the shared library can be swapped in. Its colour updates automatically to match the current state.

Usage

Icon buttons work best when space is at a premium and the icon meaning is instantly clear. When in doubt, use a labelled Button instead.

✔ Use when
  • ✓The icon alone is universally understood — close, search, add, edit, delete, overflow (⋯)
  • ✓Space is too tight for a label — toolbars, table row actions, sidebars, floating bars
  • ✓A tooltip can always be guaranteed to appear on hover and focus
✕ Avoid when
  • ✕The icon is ambiguous — users may not recognise it. Use a labelled Button instead
  • ✕It is the primary call-to-action on a screen — labels carry urgency that icons cannot
  • ✕A tooltip cannot be guaranteed — users have no text fallback

Content guidelines

Icon buttons have no visible label — the icon and its aria-label do all the communicating. Both must be immediately clear.

Choose universally recognized icons

Only use icons whose meaning is immediately clear without a label. When a suitable standard icon doesn't exist, use a labelled button instead.

Search
Delete
Close
Add
Do
Standard icons like magnifier (search), trash (delete), ✕ (close), and + (add) are understood cross-culturally.
Ambiguous
Unclear
Non-standard
Don't
Custom or decorative icons that don't map to a clear action create confusion — users will hover just to find out what they do.

Describe the action, not the icon

The aria-label is read aloud by screen readers in place of a visible label. Write it as verb + object — what will happen when the user clicks.

"Delete survey"
"Edit name"
"Close dialog"
Do
Describe the outcome — verb + object — so the label makes sense even without any surrounding context.
"Trash"
"Pencil"
"X"
Don't
Describing the icon shape ("Trash", "Pencil", "X") tells screen reader users what it looks like — not what it does.

Accessibility

→
aria-label is mandatory. Every icon button needs an aria-label describing the action, not the icon. Write aria-label="Close dialog", not aria-label="X".
→
Use a native <button> element. Gives you keyboard activation (Enter + Space) and focus management for free. Don't simulate a button with a <div> and onclick.
→
Expand the hit area on 24px. 24px falls below the 44 × 44pt minimum recommended touch target. Engineering must extend the hit area with padding or a ::before pseudo-element on touch devices.
→
Toggle buttons need aria-pressed. For bookmark, mute, or favourite states add aria-pressed="true"/"false" and update it on every interaction so assistive technology reflects the current state.
→
Always pair with a tooltip. Use role="tooltip" linked via aria-describedby. The tooltip must appear on both hover and keyboard focus — not just hover.

Installation

The Icon Button component is part of @effectory/engage-ui.

npm npm install @effectory/engage-ui 
import { EfIconButton } from '@effectory/engage-ui'
import '@effectory/engage-ui/dist/style.css' // once per app entry

Usage

Basic

<EfIconButton
  icon="plus"
  aria-label="Add member"
  @click="handleAdd"
/>

All variants

<!-- Primary: highest-emphasis, one per toolbar -->
<EfIconButton icon="plus" variant="primary" aria-label="Add member" />

<!-- Secondary: clearly discoverable supporting action -->
<EfIconButton icon="edit" variant="secondary" aria-label="Edit name" />

<!-- Tertiary: low-emphasis, use in toolbars and table rows -->
<EfIconButton icon="more-vertical" variant="tertiary" aria-label="More options" />

Sizes

<!-- 36px: default, toolbars, cards, modals -->
<EfIconButton icon="edit" size="36" aria-label="Edit" />

<!-- 24px: dense layouts, inline table actions -->
<EfIconButton icon="edit" size="24" aria-label="Edit" />

Disabled

<EfIconButton icon="plus" :disabled="!canAdd" aria-label="Add member" />

With tooltip (required)

<!-- Always wrap with EfTooltip — icon buttons have no visible label -->
<EfTooltip content="Delete survey">
  <EfIconButton
    icon="trash"
    variant="secondary"
    aria-label="Delete survey"
    @click="confirmDelete"
  />
</EfTooltip>

Props

PROPTYPEDEFAULTDESCRIPTION
icon string required Name of an Effectory icon from the shared icon library (e.g. "plus", "trash", "edit").
aria-label string required Accessible label describing the action. Write as verb + object: "Delete survey", not "Trash".
variant 'primary' | 'secondary' | 'tertiary' 'secondary' Visual emphasis level. Use one Primary per toolbar maximum.
size '36' | '24' '36' Button dimensions. '36' = 36×36px (default). '24' = 24×24px for dense layouts.
disabled boolean false Disables the button and removes it from the tab order.
type 'button' | 'submit' | 'reset' 'button' Native HTML button type. Rarely needs changing for icon buttons.

Accessibility

→
aria-label is mandatory. The component will warn in development if it is missing. Write the action, not the icon name: "Delete survey" not "Trash icon".
→
Always wrap with EfTooltip. The tooltip content should match aria-label exactly. It must show on both hover and keyboard focus — the EfTooltip component handles both automatically.
→
Expand hit area for 24px on touch. Add touch-target prop or override with CSS padding + negative margin so the tappable area meets the 44×44pt minimum on touch devices.
→
Toggle state needs aria-pressed. For bookmark, mute, or pin states pass :aria-pressed="isActive" — the component will update the attribute reactively and announce the state change to screen readers.
