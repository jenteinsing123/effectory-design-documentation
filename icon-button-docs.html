Icon Button

A fixed-size button that uses an icon instead of a text label to represent an action. Use it when space is tight and the icon's meaning is immediately clear to your audience.

🔬 Anatomy

The component has two parts. Every variant contains both layers.

① Container
② Icon
1
Container — the outer shape that defines size, background colour, border, corner radius, and shadow. All visual changes between types and states happen here.
2
Icon — the symbol inside the button. Any icon from the shared library can be swapped in. Its colour updates automatically to match the current state.

🧭 When to Use

✓ Use when
  • ✓ The icon alone is universally understood — close, search, add, edit, delete, overflow (⋯).
  • ✓ Space is too tight for a text label — toolbars, table row actions, sidebars, floating bars.
  • ✓ Supporting action alongside a clearly labelled primary action.
✗ Avoid when
  • ✗ The icon is ambiguous — users may not recognise it. Use a labelled button instead.
  • ✗ It is the primary call-to-action on a screen.
  • ✗ A tooltip cannot be guaranteed — users have no text fallback.

🧩 Variants

Three types control visual hierarchy. Choose the one that matches the importance of the action relative to its surroundings.

♿ Accessibility

→ aria-label is mandatory. Every icon button needs aria-label describing the action, not the icon. Write aria-label="Close dialog", not aria-label="X".
→ Use a native <button> element. Gives you keyboard activation (Enter + Space) and focus management for free. Don't use a <div> with onclick.
→ Touch target on 24px. 24px falls below the 44 × 44pt minimum. Engineering must extend the hit area with padding or a ::before pseudo-element.
→ Toggle buttons need aria-pressed. For bookmark, mute, or favourite buttons add aria-pressed="true/false" and update it on every interaction.
→ Always pair with a tooltip. Use role="tooltip" linked via aria-describedby. Must appear on both hover and keyboard focus.
