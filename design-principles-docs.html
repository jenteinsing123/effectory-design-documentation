Design Principles

The values that guide every design decision in the Engage product.

1. Clarity over cleverness

The UI's job is to communicate intent, not to impress.

Every interaction in Engage should be self-evident. HR managers running a survey cycle and employees responding to a pulse check are not looking for a novel experience — they want to complete a task and move on. When a pattern already exists that users recognise, use it. Novel interactions carry a learning cost that is rarely justified.

This shows up in small decisions: label buttons with verbs (Send reminder, not OK), keep navigation predictable across the Coordinator and Personal portals, and resist the temptation to collapse steps into clever micro-interactions when a straightforward flow would be clearer.

When you find yourself reaching for an unusual pattern, ask: is the complexity solving a real user problem, or is it solving an aesthetic one? If it's aesthetic, simplify.

2. Efficiency first

Reduce steps, surface the right action at the right moment, and never make users think twice.

Users of the Engage product are completing tasks inside a broader workday. A coordinator setting up a survey has a deadline. An employee answering a check-in has two minutes between meetings. Neither is exploring the interface for its own sake.

Design for the critical path. Default values should reflect the most common choice. Destructive or irreversible actions should require confirmation, but routine actions should not add friction. Progressive disclosure is your friend: surface essential controls upfront and move advanced options behind a secondary layer.

Measure efficiency not just in click counts but in cognitive load. A page with three prominent calls-to-action is often slower to use than a page with one, even if the one-CTA page technically requires more clicks. Clarity and efficiency compound each other.

3. Accessible by default

WCAG 2.1 AA is the floor, not the ceiling.

Every component in the design system ships with keyboard navigation, visible focus states, appropriate ARIA roles and labels, and colour contrast that meets WCAG 2.1 AA. Accessibility is built into the component, not bolted on during a compliance audit.

This matters especially in an employee feedback context. Engage is used across entire organisations — which means users with visual impairments, motor limitations, or reliance on assistive technology will encounter these interfaces. Excluding them is not a permissible trade-off.

In practice: treat accessibility as a constraint that shapes design decisions from the start, not as a checklist to pass at the end. If a proposed interaction cannot be made keyboard-accessible without awkward workarounds, that is a signal to revisit the interaction itself. The system's token set ensures all text and interactive elements meet contrast requirements out of the box — never override colour tokens with raw values that break this guarantee.

4. Consistent, not rigid

The system provides a foundation, not a cage.

Consistency reduces the cognitive overhead of learning a new part of the product — users carry mental models from one screen to the next. But consistency applied without judgment produces interfaces that technically follow the rules while failing the people using them. Context matters.

Components are composable. Tokens are overridable at the portal level (Coordinator and Personal each have their own brand colour layer). A component that needs to behave differently in a specific context can do so — but the divergence should be intentional, documented, and ideally fed back as a variant proposal to the DS team so that others can benefit.

The right question is not "does this follow the system?" but "does this serve the user while remaining coherent with the rest of the product?" If the answer requires bending a guideline, document why. Undocumented divergence becomes technical debt; intentional, documented divergence becomes a system contribution.

5. Trust through transparency

Feedback products handle sensitive data. The UI must never surprise users.

Engage collects employee feedback — sometimes anonymised, sometimes sensitive. Users need to trust the product. Trust is built through transparency: the interface must be honest about what is happening, what data is being collected, and what actions are irreversible.

Concretely, this means the system must communicate all meaningful states. Loading must look like loading (not a blank page). Errors must explain what went wrong and what the user can do next — not just show a red border and a generic message. Empty states should explain why content is absent and what action creates it. Partial data (e.g. a report based on incomplete responses) must be labelled as such.

Destructive actions — deleting a survey, removing a participant, closing a feedback cycle — require explicit confirmation. The confirm dialog must name the specific thing being destroyed and the consequences. "Are you sure?" is not enough; "Delete Q2 Engagement Survey? This cannot be undone." is.

This principle also applies to system states that are outside the user's control: if a background sync is in progress, show it. If a submission succeeded, confirm it clearly and immediately. Users who are uncertain whether their action registered will act again — often at the wrong moment.
