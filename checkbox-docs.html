Checkbox

A binary toggle for selecting one or more options from a set. Four variants cover everything from a bare control in a table cell to a full card button in a filter bar.

Anatomy

The bare checkbox is three layers — a container, a selection indicator (check or dash), and an optional border ring. The label, description, and button variants wrap this control with additional content.

1Container
2Indicator
1
Container — 16×16 px square (20×20 in large size), 4 px radius. Unchecked: white background + 1.5 px border/action border. Checked or indeterminate: filled bg/brand/base with no visible border.
2
Indicator — a white check SVG when checked, a white dash SVG when indeterminate. In the "select all" variant the container stays white and the indicator is teal instead of white. Hidden when unchecked.

Usage

Checkboxes let users make independent binary choices. Use them when selections are non-exclusive — multiple items can be chosen at once. For a single on/off toggle, use the Toggle instead.

Use when
  • Users can select zero, one, or many items from a list
  • Each option is independent — selecting one doesn't affect others
  • Options need a label and optionally a short description
  • You're building a filter row and need compact togglable chips
Avoid when
  • Only one option can be active at a time — use Radio Group
  • The action takes effect immediately without a "Save" step — use Toggle
  • You have more than ~8 options — consider a multi-select Select

Choosing a variant

VARIANTWHEN TO USE
CheckboxTable cells, dense lists, custom layouts where you supply your own label. Comes in default (16 px) and large (20 px).
Checkbox with labelStandard form fields and settings lists. The label is always to the right; check left-aligns with the first line of text so wrapping labels stay readable.
Info checkboxWhen each option needs a short description or a "Learn more" link — onboarding flows, survey builders, plan selection. Has a responsive desktop/mobile layout.
Checkbox buttonCompact horizontal filter bars or tag-picker rows where the entire pill surface is the tap target. Combines a checkbox indicator with a full-bleed clickable area.

Best practices

Do
Give every checkbox a visible text label that describes exactly what selecting it does.
Don't
Never render checkboxes without labels — screen readers will announce the control with no context, and sighted users won't know what they're selecting.
Do
Show the indeterminate state on a "select all" checkbox when only some children are selected — it tells users the list is partially checked.
Don't
Don't leave the parent checkbox unchecked when children are partially selected — the state is ambiguous and users can't tell whether any items are chosen.

Variants

Four variants cover the full range of checkbox use cases — from bare controls in data tables to full-surface button chips in filter bars.

Checkbox — the bare control. Default size is 16×16 px; Large is 20×20 px. Use in table cells, custom layouts, or anywhere you supply your own label markup.
Checkbox with label — wraps the control with a right-aligned text label. The label and checkbox share a single click target — always use a semantic <label> element.
Label Subtitle with a short explanation. Learn more ↗
Info checkbox — adds a secondary description line and optional "Learn more" link below the label. Responds to desktop and mobile widths.
Checkbox button — a pill-shaped card where the entire surface is the tap target. Used in compact horizontal filter bars or tag pickers. The embedded checkbox indicator shows selection state; you don't need a separate label element outside the button.

Accessibility

→
Always use a native <input type="checkbox">. You get keyboard focus, Space to toggle, and correct announcements ("checkbox, checked") for free.
→
Link every checkbox to its label. Use a wrapping <label> or pair id on the input with for on the label — both expand the tap target and make the association audible to screen readers.
→
Programmatic indeterminate only. The indeterminate state has no HTML attribute — set it in JavaScript with input.indeterminate = true. Screen readers announce it as "mixed".
→
Group related checkboxes in a <fieldset> with a <legend>. The legend becomes the group label that screen readers read before each option.
→
Checkbox button still needs a semantic input. The <label> pill wraps a real <input type="checkbox"> — don't replace it with a <div role="checkbox">. The native element handles keyboard and AT for free.

Usage

Bare checkbox

<!-- Unchecked -->
<input type="checkbox" class="cb" id="option-1" name="option-1" />

<!-- Checked -->
<input type="checkbox" class="cb" id="option-2" name="option-2" checked />

<!-- Indeterminate (set via JS) -->
<input type="checkbox" class="cb" id="select-all" />
<script>
  document.getElementById('select-all').indeterminate = true;
</script>

<!-- Large size -->
<input type="checkbox" class="cb cb-lg" id="option-3" />

Checkbox with label

<label class="cb-label-wrap">
  <input type="checkbox" class="cb" name="summary" checked />
  Send me weekly summary emails
</label>

<!-- Disabled -->
<label class="cb-label-wrap is-disabled">
  <input type="checkbox" class="cb" disabled />
  Disabled option
</label>

Info checkbox

<div class="cb-info-wrap">
  <input type="checkbox" class="cb" id="visibility" checked />
  <label class="cb-info-body" for="visibility">
    <span class="cb-info-label">Manager visibility</span>
    <span class="cb-info-desc">
      Managers can view their team's anonymised results.
      <a href="/help/visibility" class="link-inline"
         target="_blank" rel="noopener noreferrer">Learn more<span aria-hidden="true">&nbsp;↗</span></a>
    </span>
  </label>
</div>

Checkbox button

The entire <label> surface is the click target. The nested <input> drives state; pointer-events: none stops it from capturing its own clicks.

<label class="cb-btn">
  <input type="checkbox" class="cb" name="digest" />
  Weekly digest
</label>

<!-- Disabled -->
<label class="cb-btn is-disabled">
  <input type="checkbox" class="cb is-disabled" disabled />
  Alerts
</label>

Select all with indeterminate

<label class="cb-label-wrap">
  <input type="checkbox" class="cb" id="select-all" />
  All surveys
</label>
<script>
  // Set indeterminate when some (not all) children are checked
  document.getElementById('select-all').indeterminate = true;
</script>

Accessibility

→
Always use <input type="checkbox">. Never substitute a <div role="checkbox"> — the native element provides keyboard interaction, AT announcements, and form participation for free.
→
Wrap in <label> or use for/id. Either approach links the visible text to the input so screen readers announce the label when the control is focused.
→
Indeterminate requires JavaScript. There is no HTML attribute for the indeterminate state — set it with el.indeterminate = true. Screen readers will announce it as "mixed".
→
Group with <fieldset> + <legend>. The legend names the group — screen readers read it before each option so users always know what they're selecting.
→
Add disabled to the <input>, not just the wrapper. The native attribute removes it from the tab order and prevents form submission. The .is-disabled class handles the visual only.
