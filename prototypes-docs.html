Prototype System

Build realistic product screens using the real components and design tokens of this system — directly from Claude Code.

What is it

A Claude Code skill that generates HTML prototypes using the exact same CSS, tokens, and icons as the real product.

The Effectory Design System skill knows every component, variant, token, and icon in this system. When you ask Claude to build a screen, it reads from a reference that is kept in sync with this documentation. Prototypes look like the real product because they are the real product's styles — not an approximation built from scratch.

Generated prototypes are saved in the prototypes/ folder and can be opened in a browser immediately via a local server.

One-time setup

Three steps — only needed once per machine.

1
Clone (or pull) the repo
The skill lives inside the repo at .claude/skills/effectory-design-system/. Clone it once, then stay up to date with git pull before each session.
2
Install Claude Code
Install via npm: npm install -g @anthropic-ai/claude-code. Requires Node 18 or higher.
3
Start Claude Code from the repo root
cd <repo> && claude — Claude Code picks up the skill automatically from .claude/skills/.

Building a prototype

Start each session with a pull so you always have the latest components and skill.

1
Pull before you start
Components, tokens, and the skill reference are all in this repo. A git pull keeps everything in sync before you build.
2
Ask Claude to build the screen
Describe the screen naturally. The skill triggers on words like prototype, design, scherm, and pagina. Examples:
"bouw een prototype van het instellingenscherm" "maak een design voor een onboarding-flow met drie stappen" "prototype van een dashboard met filters en een tabel"
3
Claude saves the file and tells you the URL
The skill generates an HTML file in prototypes/, only using components and tokens that exist in this system. If something is missing, it stops and reports it rather than inventing a workaround.

Viewing the prototype

Always open prototypes via the local server — never by double-clicking the file.

⚠️
Do not open via double-click (file://). SVG masks (e.g. the toggle checkmark) and icon loading rely on HTTP. Opening via file:// silently breaks icons and some visual component details — the prototype looks incomplete, but the skill is fine.
1
Start the local server
In the repo root: python3 serve.py. The server prints the port it's running on.
2
Open the URL Claude provides
After building, Claude always prints the exact URL: http://localhost:<port>/prototypes/<filename>.html. Paste that into your browser.

Ground rules

The skill enforces the design system rather than working around it.

Only existing components, tokens and icons. The skill reads directly from this documentation's CSS and token files. It never invents class names, color values, or variants.
Missing component? Claude will tell you. If a requested component doesn't exist in the system, the skill stops and reports it explicitly — it never silently builds a substitute.
No doc-site chrome. Prototypes are product screens, not documentation pages. The skill never adds the sidebar, page tabs, or TOC from this site to a prototype output.
Tokens, not hardcoded values. All spacing, color, and shadow values come from design tokens. Prototypes automatically inherit dark mode and portal theming (Coordinator / Personal).

Keeping the skill up to date

For those who maintain the design system — three files need to stay in sync with the documentation.

File Update when
components.css A component is added or its CSS changes. Bundle the updated <style> block from the component's doc page into this file.
foundation.css Spacing, radius, shadow, or typography variables change. Mirrors the :root values from styles.css that prototypes need.
.claude/skills/effectory-design-system/design-system-reference.md A new component, variant, token, or icon is added. This is the reference Claude reads before building — if it's not listed here, Claude won't use it.
tokens.css Auto-generated only. Run python3 build-tokens.py after updating tokens in Figma. Never edit this file by hand.

If these files drift out of sync with the documentation, the skill will build prototypes with outdated or missing components without any warning. A git pull before each session is the simplest safeguard.
